Gate Semanal — Lógica de Progressão

Após entrar no app, o progresso é controlado por um sistema de gates semanais:

🔒

Locked (padrão)

Semanas 2–24 começam bloqueadas. Conteúdo visível na Biblioteca mas não interativo.

▶

Available (semana ativa)

Semana 1 começa disponível. Usuário faz check-in diário (máx. 1 por dia).

✓

Avanço de Semana

Após 5 check-ins, botão "Avançar" fica disponível → POST /gate/advance → próxima semana desbloqueia.

✦ Geração do Plano de Ação (IA)

Core Feature
POST /plano/generate

Após o onboarding, o usuário clica em "Gerar o meu plano de ação". O backend chama o Claude Sonnet com o framework ALMA v5.14 + dados do negócio para gerar um plano estratégico de 24 semanas completamente personalizado.

  • Modelo: Claude Sonnet 4.6 · Output: até 32.000 tokens
  • Context: ALMA v5.14 (214K chars, prompt caching ativado)
  • Output: 24 semanas · nome, objetivo, por que importa, versículo, 2–4 tarefas, indicador, resultado
  • Idempotente: se plano já existe, retorna o existente
  • Tempo estimado: ~4–5 minutos (timeout do app: 2 minutos)

Bugs Identificados e Corrigidos

Análise e resolução dos problemas encontrados no fluxo de geração do plano

Foram identificados 3 bugs na geração do plano. Todos foram corrigidos nesta sessão. Os bugs foram descobertos executando uma simulação end-to-end idêntica ao código de produção.

✓

Bug #1 — ALMA não copiado no build (causa raiz do "Erro interno do servidor.")

Corrigido
Causa

O comando de build (tsc) só compila arquivos .ts. O arquivo alma-content.txt (214KB, framework estratégico) não era copiado para dist/lib/. O servidor de produção no Render roda node dist/server.js — ao tentar ler o ALMA, lançava erro ENOENT (arquivo não encontrado) antes mesmo de chamar a IA. Como não havia try-catch, o erro caía no handler global → "Erro interno do servidor."

Correção

package.json — script build:

tsc && cp src/lib/alma-content.txt dist/lib/alma-content.txt

service.ts: Envolveu loadAlmaContent() em try-catch, lançando ALMA_LOAD_ERROR com mensagem descritiva.

✓

Bug #2 — MAX_TOKENS insuficiente para 24 semanas completas

Corrigido
Causa

O limite de output era MAX_TOKENS=16.000. A simulação de produção confirmou: o modelo gera exatamente 16.000 tokens e é cortado no meio da semana 24 (JSON sem fechar). O JSON.parse() falha com "Unterminated string". Este seria o segundo erro revelado após o Bug #1 ser corrigido.

Correção

Aumentado MAX_TOKENS de 16.000 para 32.000.

Dados da simulação: input=1.824 tokens (user msg) + ~53K (ALMA cached) · output=16.000 tokens cortados em 50.720 chars · plano incompleto na semana 24.

✓

Bug #3 — FK violation: acao_codigo gerado pela IA com valor inválido

Corrigido
Causa

O prompt pede à IA para retornar "acao_codigo": null, mas o modelo às vezes retorna valores inventados (ex: "LID-07"). Como acoes_library está vazia em produção, o INSERT INTO tarefas_semana falharia com violação de chave estrangeira dentro da transação de persistência.

Correção

No INSERT de tarefas, o valor de acao_codigo é agora sempre forçado para null, independente do que a IA retornar.

Erro sem tratamento (Anthropic SDK)

✓

Chamada à API Anthropic sem try-catch

Corrigido
Causa

A chamada client.messages.create() não estava em try-catch. Qualquer erro de rede, autenticação ou limite da API gerava um erro bruto (não AppError) que caía no handler genérico, mascarando a causa real.

Correção

Chamada envolvida em try-catch. Lança AppError com código AI_CALL_ERROR e mensagem descritiva incluindo o status HTTP e erro real da API.

Ação Necessária — Deploy

Para ativar as correções em produção: É necessário fazer um novo deploy no Render. O primeiro npm run build irá executar automaticamente o cp alma-content.txt e a geração de planos passará a funcionar. Os 3 bugs são corrigidos com este único deploy.

Observações para o PO

  • ⚠
    Tempo de geração do plano (~4–5 min): O timeout atual do app é 2 minutos. Após o deploy, testar se o plano completo é gerado dentro desse tempo ou se é necessário aumentar o timeout ou usar streaming.
  • ⚠
    Status da assinatura no Perfil: Atualmente exibido como "Activa" de forma estática. Pode ser dinamizado consultando GET /subscription/status.
  • ✓
    Idempotência: O plano é gerado uma única vez por usuário. Chamadas subsequentes ao POST /plano/generate retornam o plano existente sem custo de IA adicional.
  • ✓
    Prompt Caching: O conteúdo ALMA (214KB) é enviado com cache_control: ephemeral. A partir da segunda chamada, a leitura do cache reduz custo e latência em ~85%.
